# 驾驶舱全表统计优化

**日期**：2026-04-26  
**范围**：非遗模块首页驾驶舱（Dashboard）中「展品总数」统计方式。

## 背景与问题

驾驶舱方法 `IHeritageItemService.getDashboardData()` 在缓存未命中时，曾通过 `heritageItemMapper.selectHeritageItemList(null).size()` 得到「总展品数」。

- `selectHeritageItemList` 会按列表查询条件拉取**整表有效行**（`del_flag = '0'`，无名称/分类/状态筛选时即全量未删数据）到 JVM，再取 `List` 元素个数。
- 随展品数量增长，会产生大结果集扫描、高内存与 GC 压力，存在 OOM 与慢查询风险，与同一方法内已使用的 `sumViewCount()`、`selectCategoryStats()` 等聚合查询风格不一致。

## 优化方案

1. 在 `HeritageItemMapper` 中新增 `countValidHeritageItems()`，语义与「无入参的列表查询」一致：仅统计 `heritage_item` 中 `del_flag = '0'` 的行数。
2. 在 `HeritageItemMapper.xml` 中实现为单条 SQL：`SELECT COUNT(*) FROM heritage_item WHERE del_flag = '0'`。
3. 在 `HeritageItemServiceImpl.getDashboardData()` 中用上述返回值设置 `IndexStatsVo.totalItems`，不再执行全表 `select` + `size()`。

## 涉及文件

| 文件 | 变更说明 |
|------|----------|
| `ruoyi-admin/.../HeritageItemMapper.java` | 声明 `countValidHeritageItems()` |
| `ruoyi-admin/.../HeritageItemMapper.xml` | 增加 `countValidHeritageItems` 的 SQL 映射 |
| `ruoyi-admin/.../HeritageItemServiceImpl.java` | 用聚合结果设置 `totalItems` |

## 功能与数据一致性

- 与修改前 `selectHeritageItemList(null)` 的**计数语义**一致：只统计未逻辑删除的展品，列表侧同样固定 `and del_flag = '0'`，无额外动态条件时条数与 `COUNT(*)` 相同。
- 仍受原有 Redis 驾驶舱缓存（`heritage:stats:dashboard`）与双重检查锁逻辑约束，仅替换「取总数」这一步骤的底层实现。

## 验证建议

1. 数据库执行：`SELECT COUNT(*) FROM heritage_item WHERE del_flag = '0'`，与接口 `GET /heritage/stats/dashboard` 返回的 `totalItems` 对比（注意清除或等待驾驶舱缓存过期后再测）。
2. 打开 MyBatis SQL 日志时，刷新驾驶舱应只出现 `COUNT(*)` 相关语句，不再出现为统计条数而执行的无 `LIMIT` 的全列 `SELECT ... FROM heritage_item`。

- 生产环境凭据、Druid/Swagger 访问控制等安全项未改动（个人展示场景可另排期）。
