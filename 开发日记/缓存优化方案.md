
# 高性能缓存优化方案技术梳理

## 一、 核心痛点分析

在原有的业务代码中，虽然引入了 Redis，但仍存在以下高并发场景下的性能隐患：

1.  **数据库写压力过大（热点行锁竞争）**：
    *   **现象**：每次用户浏览展品，都会触发一次数据库 `UPDATE` 操作。
    *   **后果**：如果有 1000 人同时浏览同一个热门展品，MySQL 会对同一行记录加排他锁（X锁）。大量线程阻塞等待锁释放，导致数据库连接池耗尽，甚至拖垮整个系统。
2.  **缓存击穿风险**：
    *   **现象**：大屏驾驶舱的聚合查询（`getDashboardData`）非常耗时（涉及多表 Count/Sum）。当缓存 Key 过期的一瞬间，如果正好遭遇高并发请求。
    *   **后果**：大量请求瞬间穿透 Redis 直击数据库，可能导致数据库 CPU 飙升甚至宕机。
3.  **缓存一致性与粒度问题**：
    *   **现象**：原本的设计中，只要有点赞/浏览，就清除大屏缓存。
    *   **后果**：在高频互动下，大屏缓存频繁失效，缓存形同虚设（Cache Thrashing）。

---

## 二、 优化方案详解

针对上述问题，我们采用了 **多级缓存 + 写缓冲（Write-Behind） + 互斥锁** 的组合策略。

### 1. 浏览量计数优化：Redis 原子计数 + 异步落库

**优化目标**：解决热点数据的数据库行锁冲突问题。

*   **实现方法 (`incrementCacheMapValue` + `addViewCount`)**：
    *   **写操作**：用户浏览时，不再直接操作数据库。而是调用 `RedisCache.incrementCacheMapValue`，利用 Redis 的 `HINCRBY` 命令，在内存中对 `heritage:view_buffer` 这个 Hash 结构的对应 ItemId 字段进行原子 +1。
    *   **底层原理**：Redis 是单线程模型（核心命令执行），天然支持原子性操作，不需要加锁，且完全是在内存中操作，性能是 MySQL 的数万倍。
*   **读操作 (`selectHeritageItemByItemId`)**：
    *   在查询展品详情时，采用 **“读时合并”** 策略。
    *   `最终展示浏览量` = `MySQL中的持久化基数` + `Redis中的缓冲增量`。
    *   这保证了即使用户没有实时看到数据库更新，他看到的数字也是实时准确的。

*   **数据持久化 (`HeritageTask` 定时任务)**：
    *   **逻辑**：创建一个 Quartz 定时任务，每 10 分钟执行一次。
    *   **流程**：
        1.  从 Redis 取出所有展品的浏览增量 (`getCacheMap`)。
        2.  遍历 Map，执行 SQL 更新数据库 (`update set view = view + #{delta}`)。
        3.  **关键点**：更新成功后，使用负数增量 (`HINCRBY key field -delta`) 从 Redis 中扣除已同步的数量，而不是直接删除 Key。
    *   **为什么这么做（原理）**：在“读取 Redis”到“扣除 Redis”这几毫秒的时间窗口内，可能又有用户产生了新的浏览量。如果直接 `DELETE`，这几毫秒产生的新数据就丢失了。使用原子递减扣除，可以精确保证数据最终一致性，不丢不重。

### 2. 大屏数据查询优化：双重检查锁 (Double-Checked Locking)

**优化目标**：解决高并发下的“缓存击穿”问题。

*   **实现方法 (`getDashboardData`)**：
    ```java
    // 第一次检查（无锁）
    Object cache = redis.get(key);
    if (cache != null) return cache;

    synchronized (this) {
        // 第二次检查（有锁）
        cache = redis.get(key);
        if (cache != null) return cache;

        // 查数据库 -> 写缓存
        Data data = db.query();
        redis.set(key, data);
        return data;
    }
    ```
*   **底层原理**：
    *   **第一次检查**：阻挡了 99.9% 的请求，只有缓存失效时的那几个线程能走到锁门口。
    *   **synchronized 锁**：确保同一时刻只有一个线程能去执行“查库 + 写缓存”这个重操作。
    *   **第二次检查**：这是最关键的一步。当第一个获得锁的线程执行完并释放锁后，后续排队的线程进入同步块。如果没有第二次检查，它们依然会去查数据库，导致重复查询。有了第二次检查，它们会发现“噢，前面的人已经把数据放到缓存里了”，直接拿走即可。

### 3. 展品详情查询优化：动静分离 (Dynamic-Static Separation)

**优化目标**：在保证性能的同时，解决“千人千面”的数据展示问题。

*   **问题背景**：展品的 `isLiked`（是否点赞）和 `isCollected`（是否收藏）是与当前登录用户强相关的。如果把整个对象缓存起来，A 用户看 B 用户缓存的数据，状态就全乱了。
*   **实现逻辑 (`selectHeritageItemByItemId`)**：
    *   **静态部分（缓存）**：展品的标题、图片、描述、创建时间。这些数据所有人看到的都一样，且不常变。我们将其缓存 24 小时 (`heritage:item:detail:{id}`)。
    *   **动态部分（实时查）**：
        1.  **浏览量**：如前所述，从 Redis 缓冲中取增量合并。
        2.  **用户状态**：利用 `SecurityUtils.getUserId()`，快速去 `heritage_user_action` 表（或 Redis Bitmap，视未来扩展而定）查询当前用户的交互状态。
*   **原理**：将变与不变的数据解耦。静态数据走 Redis 极速读取，动态数据通常查询很快（走索引或内存计算），两者组装后返回。

### 4. 缓存一致性维护策略

**优化目标**：平衡“数据实时性”与“系统性能”。

*   **策略调整**：
    *   **新增/修改/删除展品**：这是低频操作，但影响全局。必须**强一致性**。操作完成后，立即删除对应的详情缓存 (`deleteObject`) 和大屏统计缓存。
    *   **点赞/收藏**：这是高频操作。
        *   **详情页**：用户希望立刻看到状态变化，所以清除**单条**详情缓存，触发重查。
        *   **大屏**：大屏上的“总互动数”对实时性要求不高（差几个数无所谓）。因此，**不再清除大屏缓存**，让其自然过期（10分钟）。这避免了因为一个点赞就导致大屏缓存失效、进而引发数据库重查的“雪崩”效应。

---

## 三、 总结：优化后的性能质变

1.  **数据库减负**：浏览量写入压力减少 **99%**（假设 10 分钟内有 1000 次浏览，原本写 1000 次库，现在写 1 次）。
2.  **系统吞吐量 (QPS)**：大屏接口和详情接口完全走缓存，QPS 从数据库极限的几百提升至 Redis 极限的 **数万**。
3.  **稳定性**：消除了缓存击穿和行锁死锁的风险，系统在高并发下更加健壮。
